# 03 — Архитектура (Architecture)

## Стек и почему

| Слой | Технология | Почему |
|------|-----------|--------|
| Frontend | React + Vite + TypeScript | Быстро, типобезопасно, легко читать код |
| Стили | Tailwind CSS | Быстрый простой UI; легко заменить на UI Kit позже |
| Backend/БД | Supabase (PostgreSQL) | Готовые Auth + БД + Storage + RLS в одном месте |
| Auth | Supabase Auth | Google OAuth + email/пароль из коробки |
| Файлы (фото) | Supabase Storage | Хранение фото пользователей |
| Данные на клиенте | TanStack Query | Кеш, загрузка, обновление данных |
| Роутинг | React Router | Стандарт |
| Хостинг | Vercel | Простой деплой (позже) |

> Стек выбран под требования: роли (RLS), безопасность, скорость разработки,
> расширяемость и хорошая производительность на слабых устройствах.

---

## Принципы архитектуры

1. **Организация по фичам (feature-based).**
   Код группируется по доменам в `src/features/` (auth, users, tasks, tabs, achievements),
   а не по типам файлов. Так связанный код лежит рядом и его легко перечитывать.

2. **Маленькие файлы.**
   Один компонент / один хук / один сервис = один файл. Цель ~150–200 строк максимум.
   Это критично: Claude Code должен легко перечитывать любой файл целиком.

3. **Чёткие слои внутри фичи:**
   - `components/` — визуальные компоненты фичи
   - `hooks/` — логика и запросы (через TanStack Query)
   - `api/` — функции обращения к Supabase
   - `types.ts` — типы фичи

4. **Один Supabase-клиент** в `src/lib/supabase.ts`, импортируется везде.

5. **Безопасность на уровне БД (RLS).**
   Не полагаемся только на фронтенд. Политики Supabase Row Level Security гарантируют:
   админ видит всё, пользователь — только свои строки.

6. **Производительность:**
   - пагинация / бесконечная прокрутка для списков пользователей и задач;
   - ленивая загрузка страниц (`React.lazy`);
   - не грузим тяжёлые данные на дашборде — только сводки.

## Ключевая идея расширяемости: система вкладок

Чтобы новые вкладки добавлялись **без переписывания кода**:

1. Вкладки хранятся в таблице `tabs` (данные, не код).
2. Есть универсальный рендерер вкладки, который показывает задачи этой вкладки.
3. Новая вкладка = новая запись в `tabs`. Если ей нужны особые поля —
   они хранятся в JSONB-поле или добавляется отдельная таблица позже.

Это значит: «Медиа/Контент» — лишь первая вкладка, использующая общую модель задач.
Добавить «Продажи» или «KPI» = добавить запись в `tabs` (+ при необходимости расширить модель).

Подробная схема данных — в `docs/04-data-model.md`.
