## Что мы строим

Многопользовательская CRM-система для компании **Tijorat & Rushd**.
Пользователь = студент/бизнесмен. У каждого есть личная «карточка» (как медкарта),
где хранится его прогресс, задачи, достижения, материалы и история.

Полное описание: см. `docs/01-product-overview.md` и `docs/02-requirements.md`.

---

## Технологический стек (НЕ менять без подтверждения владельца)

- **Frontend:** React + Vite + TypeScript
- **Стилизация:** Tailwind CSS (простой UI на старте; продвинутый UI Kit придёт позже)
- **Backend / БД / Auth / Storage:** Supabase (PostgreSQL + Auth + Storage + RLS)
- **Хостинг:** Vercel (позже; пока разработка локально)
- **Роутинг:** React Router
- **Состояние/данные:** React Query (TanStack Query) для запросов к Supabase

Причины выбора — в `docs/03-architecture.md`.

## КАК МЫ РАБОТАЕМ (самое важное)

1. **Маленькими шагами.** Делаем одну фазу/задачу за раз. Не пытаемся построить всё сразу.
   Текущий план фаз — в `docs/05-roadmap.md`.
2. **Сначала читай, потом меняй.** Перед изменением файла — прочитай его и связанные файлы.
3. **Файлы держим маленькими и сфокусированными.** Один компонент = один файл.
   Если файл вырос больше ~200 строк — разбей. Это нужно, чтобы код было легко перечитывать.
4. **Перед кодом — короткий план.** Опиши в 3–6 пунктах, что собираешься сделать, потом делай.
5. **После каждой фазы — обнови `docs/07-progress.md`** (что сделано, что дальше, известные проблемы)
   и остановись для проверки владельцем.
6. **Не рефакторь всё подряд.** Меняй только то, что относится к текущей задаче.
7. **Типы — это документация.** Описывай данные через TypeScript-типы, держи их в `src/types`.
8. **Комментируй «почему», а не «что».** Код должен сам объяснять «что».
9. **Если что-то непонятно — спроси,** не угадывай молча.

Полные правила стиля кода — в `docs/06-conventions.md`.

## Принципы продукта

- **Производительность важна.** Пользователей будет много, устройства у всех разные —
  не делай тяжёлых страниц, используй ленивую загрузку и пагинацию списков.
- **Простой UI сейчас, красивый — потом.** На старте — чистый минималистичный интерфейс,
  чтобы быстро отладить сложную логику. Позже владелец даст UI Kit с цветами и анимациями.
- **Гибкая архитектура вкладок.** Новые вкладки (обучение, продажи, финансы, KPI...) должны
  добавляться через данные, БЕЗ переписывания системы. См. `docs/04-data-model.md`.
- **Безопасность через роли.** Админ видит всех; пользователь — только своё (Supabase RLS).
